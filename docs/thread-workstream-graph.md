# 线程工作流图规范

## 图的目标

单线程卡解决的是：

- 这个线程现在在干什么？

线程工作流图解决的是：

- 这个线程和别的线程是什么关系？
- 我应该先切哪个线程？
- 哪些线程只是实验分支，哪些线程才是主线？

如果用户同时开了很多线程，系统不应该要求用户把所有线程都记在脑子里。人能稳定抓住的工作记忆通常只有 5 到 7 个块，所以系统要先把几十条线程压成更少的“工作流”，再把每个工作流里的关键线程挑出来。

## 核心对象

### `workstream_id`

同一件事的工作流 id。

### `goal`

一句人话，说明这一串线程共同想解决什么。

示例：

- `把单线程恢复卡扩成可串联的线程工作图`

### `anchor_focus_name`

这条工作流最稳定的代码锚点。优先用主线线程里的具体代码符号。

示例：

- `buildThreadWorkstreamGraph`
- `CodexSidebarThreadMap.renderGraph`

### `root_thread_id`

工作流的起点线程。通常是最早定义目标的线程，不一定是当前最该看的线程。

### `nodes`

节点列表。每个节点对应一条线程，但不重复存整张卡的所有内容，只保留在图里串线必须用到的信息。

### `links`

关系边列表。边表达的不是“聊得像不像”，而是“工作上怎么接”。

### `frontier`

当前真正需要你关注的线程集合。它把线程分成三堆：

- `active_thread_ids`
  - 现在能继续做的主线或支线。
- `blocked_thread_ids`
  - 现在还得等别的线程结果。
- `parked_thread_ids`
  - 已完成、已搁置、重复、被替代，暂时不用占工作记忆。

## 允许的关系类型

### `same_goal`

两个线程服务同一目标，但不一定有严格顺序。

### `next_step`

目标线程是源线程的下一步。

示例：

- 先在 `G:\temp\codex-thread-focus-cards` 里补领域模型，再去插件目录接前端。

### `experiment_branch`

目标线程是从源线程分出去做实验的。

示例：

- 主线程在整理规则，实验线程在另一个目录里快速验证 UI 或接线方式。

### `verification_branch`

目标线程专门用来验证源线程改动是否真的可用。

### `skill_source`

源线程是技能、规范或模板来源，目标线程是在实际项目里落地它。

示例：

- 一个线程在读 `C:\Users\90778\.codex\skills\pua\SKILL.md`
- 另一个线程在把这套规则接进卡片解释层

### `workspace_handoff`

工作从一个目录或仓库接力到另一个目录或仓库。

示例：

- 在 `G:\temp` 里做原型
- 在另一个业务仓库里验证

### `blocked_by`

源线程现在卡住了，要等目标线程先出结果。

### `duplicate_of`

源线程和目标线程做的是同一件事，源线程应该尽快收掉。

### `superseded_by`

源线程已经被目标线程取代，不应该再作为主线继续看。

## 展示建议

### 工作流列表

列表视图按 `goal` 分组，每组只展示：

- `goal`
- `anchor_focus_name`
- `frontier.active_thread_ids` 的数量
- `frontier.next_suggested_thread_id`

用户先决定“进哪条工作流”，不是先在几十条线程里乱翻。

### 工作流详情

详情视图展示：

- 主线线程
- 实验分支
- 验证分支
- 技能来源
- 跨目录接力

如果边很多，默认只展开：

- `next_step`
- `blocked_by`
- `workspace_handoff`
- `skill_source`

### 线程卡和图的分工

线程卡回答“这条线程现在在做什么”。

工作流图回答“这条线程为什么存在，它和别的线程怎么接”。

两者必须同时存在，不能互相替代。
