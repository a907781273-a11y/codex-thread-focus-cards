# 问题定义

## 真实痛点

Codex 线程一多，最先丢掉的不是“历史记录”，而是“当前工作上下文”。

用户真正的问题通常不是：

- 我忘了这段对话说过什么。

而是：

- 这个线程到底在搞哪段代码？
- 现在是在查问题、改代码，还是跑验证？
- 我如果现在切回去，应该先看哪个接口、哪个方法、哪个类？

## 为什么普通摘要不够

普通大模型摘要经常会生成这种内容：

- 正在推进线程卡片能力
- 已完成部分上下文收敛
- 正在持续优化抽取逻辑

这些句子的问题不是“不够智能”，而是“没有可操作性”。

用户看到之后仍然不知道：

- 要打开哪个文件
- 要看哪个方法
- 要改哪个路由
- 要验证哪个流程

这类输出会让人持续停留在“理解线程在说什么”的阶段，而不是进入“继续做事”的阶段。

## 设计目标

线程卡片要优先回答以下问题：

1. 代码焦点是谁？
2. 当前动作是什么？
3. 下一步是什么？

换句话说，卡片不是“摘要视图”，而是“恢复心流视图”。

## 设计约束

### 1. 代码焦点必须具体

允许：

- `ThreadSummaryService.buildBrief`
- `GET /api/thread-briefs`
- `FocusResolver.resolve`
- `thread_focus_cards.status`

不允许：

- `thread/read`
- `摘要逻辑`
- `线程功能`
- `继续优化`

### 2. 只吃显式输出

为了降低幻觉和黑话污染，第一版只读取显式输出项：

- `userMessage`
- `agentMessage`
- `plan`

不使用隐藏推理内容，不把 `reasoning` 作为摘要输入。

### 3. 状态必须固定

状态不是自由文本，而是枚举值映射出的中文标签。

这样做的原因：

- 保持跨线程可比性
- 避免 GPT 风格漂移
- 避免把“动作”写成“态度”

## 成功标准

如果用户隔一段时间切回某个线程，应该在 5 秒内回答：

- 这个线程现在在做什么？
- 它和哪个具体代码符号相关？
- 我下一步该干什么？
