# 问题定义

## 真实痛点

Codex 线程一多，最先丢掉的不是“历史记录”，而是“当前工作上下文”。

用户真正的问题通常不是：

- 我忘了这段对话说过什么。

而是：

- 这个线程到底在搞哪段代码？
- 现在是在查问题、改代码，还是跑验证？
- 我如果现在切回去，应该先看哪个接口、哪个方法、哪个类？

当线程继续增加后，还会冒出第二层问题：

- 这条线程是主线，还是实验分支？
- 这条线程是上一步，还是下一步？
- 这条线程只是读 skill，还是已经进入业务仓库验证？

## 为什么普通摘要不够

普通大模型摘要经常会生成这种内容：

- 正在推进线程卡片能力
- 已完成部分上下文收敛
- 正在持续优化抽取逻辑

这些句子的问题不是“不够智能”，而是“没有可操作性”。

用户看到之后仍然不知道：

- 要打开哪个文件
- 要看哪个方法
- 要改哪个路由
- 要验证哪个流程

这类输出会让人持续停留在“理解线程在说什么”的阶段，而不是进入“继续做事”的阶段。

## 为什么单线程卡也不够

单线程卡只能解决“这条线程在做什么”，还解决不了“这些线程怎么接”。

真实工作里经常会出现这些情况：

- 一个线程负责拆规则
- 一个线程负责写原型
- 一个线程负责在另一个目录里做实验
- 一个线程负责跑真实验证

如果系统不把这些关系串起来，用户就得自己在脑子里维护一张图。问题是人的工作记忆通常只能稳定抓住 5 到 7 个块，线程一多，这张图就会立刻散掉。

所以第二层目标不是继续写更长的摘要，而是把线程压缩成“工作流”，再告诉用户主线在哪、分支在哪、下一步该切哪条线程。

## 设计目标

线程卡片要优先回答以下问题：

1. 代码焦点是谁？
2. 当前动作是什么？
3. 下一步是什么？
4. 它和别的线程是什么关系？

换句话说，卡片不是“摘要视图”，而是“恢复心流视图”。

## 设计约束

### 1. 代码焦点必须具体

允许：

- `ThreadSummaryService.buildBrief`
- `GET /api/thread-briefs`
- `FocusResolver.resolve`
- `thread_focus_cards.status`

不允许：

- `thread/read`
- `摘要逻辑`
- `线程功能`
- `继续优化`

### 2. 只吃显式输出

为了降低幻觉和黑话污染，第一版只读取显式输出项：

- `userMessage`
- `agentMessage`
- `plan`

不使用隐藏推理内容，不把 `reasoning` 作为摘要输入。

### 3. 状态必须固定

状态不是自由文本，而是枚举值映射出的中文标签。

这样做的原因：

- 保持跨线程可比性
- 避免 GPT 风格漂移
- 避免把“动作”写成“态度”

## 成功标准

如果用户隔一段时间切回某个线程，应该在 5 秒内回答：

- 这个线程现在在做什么？
- 它和哪个具体代码符号相关？
- 我下一步该干什么？
- 它属于哪条工作流？
- 我是继续它，还是切去它的下一步线程？
